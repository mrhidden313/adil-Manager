## 1. VISION & MISSION
AK FLOW is a B2B SaaS platform designed to manage multiple tenant agencies. It allows the Super Admin to onboard "Agencies" (Companies), assign Managers to them, and let those Managers handle their own Sales and Fulfillment teams.
The core workflow: Sales agents submit Tickets, and Fulfillment agents process and complete them.

## 2. TECH STACK & INFRASTRUCTURE
* **Frontend:** SvelteKit (Svelte 5), TailwindCSS v4, Vite PWA.
  * *Deployment:* Hosted on Vercel (`akflow.vercel.app`).
  * *Architecture:* Pure Single Page Application (SPA). Server-Side Rendering (SSR) is completely DISABLED to prevent `sessionStorage` errors.
* **Backend:** Node.js, Express, TypeScript, Prisma ORM.
  * *Deployment:* Hosted on an Ubuntu VPS (IP: `69.164.245.230`) using PM2 (process manager).
  * *Web Server:* Nginx acts as a reverse proxy, routing traffic from Port 443 (HTTPS) to Port 3000 (Node.js).
  * *SSL:* Let's Encrypt / Certbot is used for the domain `akflow.duckdns.org`.
* **Database:** PostgreSQL (Hosted on Supabase).
* **File Storage:** Supabase Storage bucket named `proofs` (used for payment proof screenshots and profile pictures).

## 3. AUTHENTICATION & SECURITY
* **System:** Custom JWT-based stateless authentication.
* **Roles:** SUPER_ADMIN, MANAGER, SALES, FULFILLMENT.
* **Super Admin Login:** adminadil@akflow.com / adill1122@@@
* **Critical Security Patch:** The JWT middleware (`auth.middleware.ts`) was specifically modified to query the database on every request to ensure the user still exists and is `isActive`. This was done to prevent deleted users from retaining access via unexpired JWT tokens.

## 6. HOW TO APPLY BACKEND CHANGES (Step by Step)

AI makes changes locally → User runs commands on VPS via SSH

### Step 1: AI makes code changes locally and pushes to GitHub
The AI edits files in `c:\Users\salar pc\Documents\Adil manager\` and runs:
  git add .
  git commit -m "description of changes"
  git push

### Step 2: User SSHs into VPS and runs these commands
  cd ~/adil-Manager/backend

  git pull                                      # Pull latest code from GitHub
  npm install                                   # Install any new dependencies
  npx prisma db push --accept-data-loss         # ONLY if schema.prisma changed
  npx prisma generate                           # ONLY if schema.prisma changed
  npm run build                                 # CRITICAL: compile TypeScript to JS
  pm2 restart backend                           # Restart the live server

### If PM2 says "Process not found" (first-time or after server reboot):
  pm2 start dist/index.js --name backend        # Start fresh
  pm2 save                                      # Save process list for auto-restart
  pm2 startup                                   # Make PM2 auto-start on boot (run once)

### Useful PM2 Commands:
  pm2 status           # See if backend is running (should say 'online')
  pm2 logs backend     # See live backend logs (errors, crashes)
  pm2 restart backend  # Restart after code changes
  pm2 stop backend     # Stop the server

### Step 3: Frontend changes are automatic
The frontend is on Vercel. When AI pushes to GitHub, Vercel auto-detects the
change and rebuilds within 1-2 minutes. No manual action needed by the user.

## 7. CRITICAL PROBLEMS FACED & SOLVED
If you are an AI reading this, DO NOT revert these fixes:

1. **Vercel SSR 500 Errors:** Frontend uses sessionStorage which is undefined in Node.js.
   Solution: `src/routes/+layout.ts` has `export const ssr = false;` — DO NOT remove this.

2. **Missing Svelte Imports:** Un-imported capitalized components crash Vercel builds.
   Solution: Always explicitly import every component used in a Svelte file.

3. **Nginx WebSocket Config:** Socket.IO requires special Upgrade headers in Nginx config.
   The proxy_set_header Upgrade and Connection headers must be present.

4. **TypeScript Build Errors:** npm run build (tsc) MUST succeed before pm2 restart.
   If build fails, the old dist/index.js keeps running with old code. Fix TS errors first.

5. **PM2 "Process not found":** Happens after server reboots. Use pm2 start instead of
   pm2 restart. Then run pm2 save to persist the process list.

6. **Prisma Client Mismatch:** After schema changes, always run npx prisma generate on
   the VPS after pushing. Otherwise the old client types will be used even with new DB schema.

## 9. BUSINESS LOGIC
* **Ticket System:** Sales agents create Tickets with a `price` field. Each ticket
  automatically gets `bonusAmount = price * 0.10` (10% commission for the sales agent).
* **Payout System:** Manager reviews completed tickets with bonusStatus=PENDING, then
  triggers a Payout by uploading a proof screenshot. The agent approves it from their dashboard.
* **Multi-tenancy:** Each Company (Agency) is isolated. Managers only see their own team.
  Super Admin can see everything and impersonate any company.
* **Chat:** Real-time team chat via Socket.IO, scoped per company. Can be disabled per company.
